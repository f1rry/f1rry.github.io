Spring学习 Spring MVC基础学习

简介

Spring MVC基于模型-视图-控制器模式实现，它能够帮你构建像Spring框架那样灵活和松耦合的Web应用程序。
一个由Spring MVC处理的请求基本流程如下

其中DispatcherServlet是一个前端控制器，在这里有个单实例的Servlet将请求委托给应用程序的其他组件来执行实际的处理。它的任务是将请求发送给Spring MVC控制器。

由于控制器会有多个,DispatcherServlet需要知道应用将请求发送给那个控制器。所以DispatcherServlet会查询一个或多个处理器映射(handler mapping来确定请求的下一站)

处理器映射会根据请求所携带的URL信息来进行决策，选择适当的控制器来处理。

控制器在处理完请求后，会返回一些用于返回给用户的信息(被称为模型)以及用于展示这些信息的视图的逻辑视图名

DispatcherServlet会根据逻辑视图名在视图解析器中查找相应视图，然后再将模型数据传递给特定视图，最后由视图进行响应

基础配置

DispatcherServlet配置

DispatcherServlet配置的方法一般有两种:

  • 传统方式，DispatcherServlet作为一个Servlet，可以在web.xml文件中配置
  • 借助于Servlet 3规范和Spring 3.1的功能增强，可以使用Java将DispatcherServlet配置在Servlet容器中，而不再使用web.xml文件

使用Java将DispatcherServlet配置在Servlet容器中

package spitter.config;
import  org.springframework.web.servlet.support.AbstractAnnotationConfigDispatcherServletInitializer;

public class SpitterWebAppInitializer extends AbstractAnnotationConfigDispatcherServletInitializer{
    @Override
    protected  String[] getServletMappings() {

        /* 将DispatcherServlet映射到"/",使所有的请求都经过DispatcherServlet */
        return new String[] { "/" };
    }

    @Override
    protected Class<?>[] getRootConfigClasses() {
        return new Class<?>[] { RootConfig.class };
    }

    /*与DispatcherServlet有关，指定配置类*/
    @Override
    protected Class<?>[] getServletConfigClasses() {
        return new Class<?>[] { WebConfig.class };
    }
}

在具体描述上述代码之前，我们先说说Servlet容器的配置过程。
在Servlet3.0环境中，

  1. 容器会在类路径中查找实现 javax.servlet.ServletContainerInitializer 接口的类，如果能发现的话，就会用来配置Servlet容器。
  2. Spring提供了这个接口的实现，名为SpringServletContainerInitializer，这个类反过来又会查找WebApplicationInitializer的类并将配置的任务交给它们来完成。
  3. Spring3.2引入了一个遍历的WebApplicationInitializer基础实现，也就是AbstractAnnotationConfigDispatcherServletInitializer

因为上述的类继承了 AbstractAnnotationConfigDispatcherServletInitializer 因此当部署到 Servlet3.0容器中的时候，容器会自动发现它，并用它来配置Servlet上下文

注意到我们自己写的类继承了 AbstractAnnotationConfigDispatcherServletInitializer，并且重写了它的三个方法

  • getServletMappings() 他会将一个或多个路径映射到DispatcherServlet上
  • getRootConfigClasses()
  • getServletConfigClasses()

除此之外它还有个方法会较多地用于重写

Spring Web应用有两个应用上下文，它们分别是：

  • 由DispatcherServlet创建
    通过getServletConfigClasses()方法来指定加载应用上下文时，使用哪个配置类中的bean。
    加载包含Web组件的bean,如控制器、视图解析器、处理器映射

  • 由ContextLoaderListener创建
    通过getRootConfigClasses()方法来指定加载应用上下文时，使用哪个配置类中的bean。
    加载应用中的其他bean,这些bean通常是驱动应用后端的中间层和数据层组件

注意通过上述方式来配置,需要支持Servlet3.0的服务器，如Tomcat 7或更高版本。否则你就需要通过web.xml来配置了

使用web.xml配置

启用MVC

配置WebConfig

如上述配置文件所见，通过重写getServletConfigClasses()，指定了DispatcherServlet在加载应用上下文时，使用WebConfig.class配置类中的bean来加载应用中的Web组件 

@Override
protected Class<?>[] getServletConfigClasses() {
    return new Class<?>[] { WebConfig.class };
}

那么WebConfig配置类中应该包含什么内容以做到启用MVC呢 

package spitter.config;

import org.springframework.context.annotation.Configuration;
import org.springframework.web.servlet.config.annotation.EnableWebMvc;

@Configuration
@EnableWebMvc
public class WebConfig {
}

虽然这可以运行起来，它的确能够启用Spring MVC,但还有不少问题需要解决：

  • 没有配置视图解析器。如果这样的话，Spring会使用默认视图解析器，默认视图解析器会查找ID与视图名称匹配的bean,并且查找的bean要实现View接口，它以这样的方式来解析视图
  • 没有启用组件扫描。这样只能找到显示声明在配置类中的控制器。
  • 这样配置的话，DispatcherServlet 会映射为应用的默认Servlet,所以它会处理所有的请求，包括对静态资源的请求，如图片和样式表

所以我们看看最小但可用的Spring MVC配置 

package spitter.config;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.ComponentScan;
import org.springframework.context.annotation.Configuration;
import org.springframework.web.servlet.ViewResolver;
import org.springframework.web.servlet.config.annotation.EnableWebMvc;
import org.springframework.web.servlet.config.annotation.DefaultServletHandlerConfigure;
import org.springframework.web.servlet.config.annotation.WebMvcConfigureAdapter;
import org.springframework.web.servlet.view.InternalResourceViewResolver;

@Configuration
@EnableWebMvc                       //启用Spring MVC
@ComponentScan("spitter.web")       //启用组件扫描

public class WebConfig extends WebMvcConfigureAdapter {

    /*配置JSP视图解析器*/
    @Bean
    public ViewResolver viewResolver() {
        InternalResourceViewResolver resolver = new InternalResourceViewResolver();
        resolver.setPrefix("/WEB-INF/views");
        resolver.setSuffix(".jsp");
        resolver.setExposeContextBeanAsAttributes(true);
        return resolver;
    }

    /*配置静态资源的处理*/
    @Override
    public void configureDefaultServletHandling(DefaultServletHandlerConfigureer configurer){
        configurer.enable()
    }
}

注意，新的WebConfi类拓展了WebMvcConfigureAdapter并重写了其configureDefaultServletHandling()方法，通过调用DefaultServletHandlerConfigureer的enable方法，我们要求DispatcherServlet将静态资源的请求转发到Servlet容器中默认的Servlet上，而不是使用DispatcherServlet本事来处理

配置RootConfig

由于Web相关的配置通过DispatcherServlet创建的应用上下文都已经配置好了，因此现在的RootConfig相对很简单 

package spitter.config;

import org.springframework.context.annotation.ComponentScan;
import org.springframework.context.annotation.ComponentScan.Filter;
import org.springframework.context.annotation.Configuration;
import org.springframework.context.annotation.FilterType;

@Configuration
@ComponentScan(basePackages = "spitter",
    excludeFilters = {
            @Filter(type = FilterType.ANNOTATION,value = EnableWebMvc.class)
    })
public class RootConfig {
}

编写控制器

在Spring MVC中，控制器只是类上添加了 @Controller 以及方法上添加了 @RequestMapping 注解的类，@RequestMapping注解声明了它们所要处理的请求。下面是个简单的例子。

package spitter.web;

import static org.springframework.web.bind.annotation.RequestMethod.*;
import org.springframework.stereotype.Controller;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RequestMethod;

@Controller     //声明一个控制器
public class HomeController {
    @RequestMapping(value = "/" , method=GET)
    public String home(){
        return "home";
    }
}

还可以将@RequestMapping注解拆分开来 

package spitter.web;

import static org.springframework.web.bind.annotation.RequestMethod.*;
import org.springframework.stereotype.Controller;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RequestMethod;

@Controller     //声明一个控制器
@RequestMapping(value = {"/"，"/homepage"} )
public class HomeController {
    @RequestMapping(method=GET)
    public String home(){
        return "home";
    }
}

可以把@RequestMapping看做处理映射器的一部分，value属性定义了哪些url将映射到这个类或这个方法上。而method方法定义了哪些动作将映射到这个上面

接受请求的输入

Spring MVC允许以多种方式将客户端中的数据传送到控制器的处理器方法中，包括：

  • 查询参数
  • 表单参数
  • 路径参数

下面来看看分别如何处理这三种参数

  • 查询参数
    使用@RequestParam注解，注意第四行，为参数设置了默认值 

    @RequestMapping(method = RequestMethod.GET)
public List<Spittle> spittle(
    @RequestParam("max") long max,
    @RequestParam(value = "count", defaultValue="30") int count
    ){
    return spittleRepository.findSpittles(max,count)
}

  • 表单参数 

    @RequestMapping(value="/register",method = POST)
public String processRegistration(Spitter spitter){
    spitterRepository.save(spitter)
    return "redirect:/spitter/"+spitter.getUsername();
}

    这个对象有firstNme、lastName、username、password属性，这些属性将会使用请求中同名的参数进行填充
    并且注意最后一句的 redirect ，当InternalResourceViewResolver看到视图格式中的”redirect:”前缀是，它就知道要将其解析为重定向的规则，而不是视图的名称

    用于检验表单的注解，同时需要在控制器的相应参数前添加@Valid注解

}

  • 路径参数
    使用@PathVariable注解和占位符 

    @RequestMapping(value="/{spittleId}",method = RequestMethod.GET)
public String spittle(
        /*以下面这种形式，即没有value属性，则假设占位符的名称与方法的参数名相同，即{spittleId}会传递到spittleId参数中*/
        //@PathVariable long spittleId,

        /*以下面这种形式，则表明在请求路径中，{spittleId}都会传递到spittleId参数中*/
        @PathVariable("spittleId") long spittleId,
        Model model)
){
    model.addAttribute(spittleRepository.findOne(spittleId));
    return "spittle";
}

传递模型数据到视图中

传入 model 类型的参数

我们先定义一个用于数据访问的接口(为了降低耦合) 

package spitter.data;
import java.util.List;
import spitter.Spittle;
public interface SpittleRepository {
    List<Spittle> findSpittles(long max, int count);
}

然后我们创建一个实现该接口的类(到后面的jdbc,orm会讲)

最后我们再运用该类改写Controller(注意其使用了Model类作为传入参数的类型) 

package spitter.web;

import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.stereotype.Controller;
import spitter.data.SpittleRepository;
import spitter.Spittle;
import org.springframework.web.bind.annotation.RequestMapping;
import org.springframework.web.bind.annotation.RequestMethod;

@Controller
public class SpittleController {
    private SpittleRepository spittleRepository;
    @Autowired
    public SpittleController(SpittleRepository spittleRepository){
        this.spittleRepository = spittleRepository;
    }

    @RequestMapping(method = RequestMethod.GET)

    public  String spittles(Model model){
        model.addAttribute(spittleRepository.findSpittles(Long.MAX_VALUE,20));

        //model.addAttribute("spittleList",spittleRepository.findSpittles(Long.MAX_VALUE,20));
        return "spittles";
    }
}

需要注意的是spittles()方法中给定了一个Model作为参数。这样，spittles()方法就能将Repository中获取到的Spittle列表填充到模型中。Model实际上就是一个Map，它会传递给视图，这样就能渲染到客户端了。当调用addAttribute()方法并且不指定key的时候，那么key会根据值的对象类型推断确定。在本例中，因为它是一个List，因此，键会推断为spittleList。你也可以显示定义key，如上述注释方法。
